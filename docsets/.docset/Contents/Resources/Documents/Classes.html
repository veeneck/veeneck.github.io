    The purpose of this state is to wait while being attacked by a Melee unit for the first time. When engaged, this state will trigger, which causes the defender to stop moving and wait.   • IntelligenceComponent

    The state machine for a CharacterEntity. The various states and their intentions are:

    • Idle: Use minimal resources. Unit will only leave this if their leader is moving, or during a 3 second check. Last scenarion would be during an external command like attack in which case another piece of code would force the unit out of this state.

    • Leading: The squad handles the logic for most orders, but one of the units has to be the dummy leader. The leader moves to this state. Leaders have slightly different checks. For example, when moving to a point the leader will follow a path while the followers seek an agent. So for group based actions, Leading is the state for leaders. During an individual action like attacking, all units will behave the same.

    • Following: Main goal is to reach the orderedPosition of the UnitComponent. Runs logic checks to see if the unit is stuck or blocked, and moves / changes accordingly.

    • Blocked: When the leader / squad orders a unit to an invalid position (i.e.: on a physics object) the unit will transfer to blocked state and decide what to do. Options are to wait, go back to idle if leader isn’t moving, use steering to stay close by, etc.

    • Catching Up: When a unit may be stuck this state is triggered. Stuck can be distance, or contact woth an environment object. When catching up, unit will use actual pathfinding and navmesh to find its way back. This is resource intensive, so only used for edge case scenarios.

  • Ability

    Every action, command, spell, buff, etc comes down to an Ability. This class mainly serves as a container for the data while other portions of the code make the magic happen. Let’s go over some terminology which shows the ways an Ability may interact with other portions of the code.

  • Ability: Base class which contains all information about the ability including name, description, cooldown, particles and more.
  • Behavior: Enum describing the different types of abilities offered.
  • Modifier.html>Structs/Modifier.html>Modifier: An actual change an ability may incur. For example, +10 HP or +20 Speed
  • /Classes/AbilitiesComponenet.html>Classes/AbilitiesComponenet.html>AbilitiesComponenet: Entry point for processing an ability on a Classes/Squad.html>Squad. Will either run it immediately, or add to Queue via Classes/Command.html>Command.
  • /Classes/Command.html>Classes/Command.html>Command: An ability on its own will do nothing. A Classes/Command.html>Command is a container for abilities that allows Squads to work with it. A squad will execute a command either instantly, or added to its /Classes/QueueComponent.html>Classes/QueueComponent.html>QueueComponent.
  • /Structs/CommandInstructions.html>Structs/CommandInstructions.html>CommandInstructions: Abilities have no context. Structs/CommandInstructions.html>CommandInstructions provide the context by specifying target, location, etc.

  • /Classes/Action.html>Classes/Action.html>Action: A wrapper around an ability specificly for UI purposes. Actions will update icons when they are locked, cooling down, etc and give UI feedback.
