With War & Woe

Introduction

This is the main repository for the entire With War & Woe project. There are no sub modules or other configuration options needs. Just download, build and run to get started. Simulator performance is horrible, so work with a connected device. Currently requires at least an iPhone6 or iPad Air running iOS9. As of iOS9.2, performance returns to expected levels.

World Map Progression

There are a few stages for world map items. They all reference LevelData stored in the save file.

  • Has the level been revealed? If not, RevealService will be called which will animate based on the Level startPosition and endPosition. Once revealed, the save file will be edited so that a reveal never happens again for this particular level.
  • Is there travel dialogue? If so, RevealService will also take care of showing that. The duration is calculated by looking at the amount of dialogues and the amount of footsteps and returning the time for the mini cutscene. Dialogue and footsteps will only show once and not on future views.
  • Has it been completed? A completed level will have different options. A few ideas not yet implemented are for a Challenge mode to become available, the icon to change, and a high score to show.
  • Battle level vs Decision level As far as the array is concerned, a level is an icon on the map. Sometimes, an icon just links to a decision screen while others will be associated with a battle. The code splits the logic, but this explains why there are only 12 maps in the game but you see 14+ levels in the array.
  • Indicators Not implemented, but will need some sort of indicator system. Troops have upgrades available? Show red icon over Camp button. New unit acquired? Show unit Heraldry in top right of World Map. The user has to be directed throughout.
  • Achievements Certain parts of the map should trigger an achievment and animation when tapped.

The World Map is somewhat functional. Of the parts implemented, here are the major concerns: + Travel dialogue is a mess. The code needs to be fully revisited when in battle cutscenes / tutorials are implemented. + 15s Travel Music volume is too low, so the fix is hard coded right now. + Animations, scale, endPos and easing need to be finalized before fixing little glitches. + Because dialogue works with closures, memory leaks need to be test.

Abilities, Modifiers and Attributes

Ability · Modifier · AbilitiesComponent · AttributesComponent

One of the more complex parts of the code which needs to remain flexible. Let’s start with definitions.

Ability: An action carried out by a squad, character, hero or commander. Contains [Modifier] which defines the result of the ability.

Modifier: A small container that modifies any Attribute of a CharacterEntity immediately, overtime, or in intervals.

Attribute: Defining characteristic of a CharacterEntity such as Health, WalkingSpeed or Strength.

Because the line is blurred between squads, characters, commanders and heros, it is helpful to look at a few examples to see how they should flow through the code.

How does an ability get processed?

The end result of an ability is always for the [Modifier] member variable to make its way on to a target CharacterEntity. This can happen in three main ways.

  • An ability like heal from the Commander does not have an associated animation. it is instant. So, the runAbility method will set the Modifier to each unit of the target squad instantly.
  • An ability like Attack requires each character to do something over time, play an animation, and then transfer an ability to an individual target. At the time of the command, runAbility adds the command to the QueueComponent and the Ability is set to the currentAttack setting of the Character. Then, when the character processes the attack, the ability will be passed to the CombatComponent which will resolve combat and place the appropriate modifiers on the target.
  • If something physical is added to the world like an AoE fire circle, the physics delegate can process this and add the appropriate modifiers. Not implemented yet.

+5 strength to a squad when it levels up

SquadName.pList contains +5 strength modifier in appropriate level. When SquadFactory loads the squad, it will search modifiers for this level and apply them. By setting no abilityName in the pList, the code will not apply the modifier to the squads abilitiyModifiers container. Instead, it will store the modifier in [unitModifiers] which propogates the modifier down to the CharacterEntity’s ModifierComponent once a unit is added to the squad..

-10 HP to the Attack ability when a squad levels up

SquadName.pList contains -10 HP modifier in appropriate level and has the abilityName set to Attack. When the SquadFactory loads this modifier, it will be stored in the Squads abilitiyModifiers container. From here, any code that processes the ability should call AbilitiesComponent.getModifier(ability) which will return [Modifier] containing the abiltiies base modofiers combined with any squad modifiers. This function must be called in real time because live code could alter a squads capabilities.

Create a new Ability that increases walking speed. Limit use to squads not in combat

Add a new entry to Abilities.pList named Sprint. Reference other entries to see specific format. Key decisions about the ability would be to give it a behavior of .Instant which means it applies when clicked. A validation of .Move can be applied which will limit its use to moving squads. Finally, a Modifier behavior of .Duration can be applied with +WalkingSpeed which will make the unit walk faster until the time has expired.

Leveling Up

Leveling up is currently in a proof of concept state. Notes on the code flow and concerns.

Exp Granted: In ExperienceReport the results at the end of the battle are shown. All calculations are made here, but this should maybe be a dumb UI that uses a service or class. For example, exp required for the next level is pulled from SquadFactory.getExpForNextLevel which isn’t the best place for that function. Lastly, other parts of the code (like the Camp Scene) will need to show unit experience, so it shoul dbe reusable.

Calculating Unit Contribution: Currently, each modifier is tallied as a contribution point and stored in the squads BattleComponent. This is unfair to larger units with ranged attacks as they would get 10 points for every single catapult shot. Need to think through final formula which should consider damage done, damage taken, etc.

Getting New Abilities: This part does work. If the UnitName.pList file has abilities as upgrade, they will be applied to the squad if the squads exp level matches or exceeds the abilities. NOTE: Only abilities with the Automatic flag will be applied. The idea is that future support for branched choices needs to be there. A branched choice would have to cross reference the save file to see what the user selected.

Navigation

Navmesh · MeshPolygon

Navmesh is an extension of GKObstacle graph that has built in utility. MeshPolygons act as the obstacles for the graph, and also store the CGPath which allows easy checks on bounds and buffer radius. The graph is built using PhysicsEditor and exporting the results. Watch the video below for a walkthrough.
